title: Getting Started
description: Initialize a SAL project and build RDF into a local data product.
---

SAL projects are ordinary Git repositories with a `.sal` workspace for generated data.
The source files stay in your repository; the Iceberg table generated by SAL lives under `.sal/data`.

## Install SAL

Install the CLI with Go:

```sh
go install -a github.com/cgs-earth/sal@latest
```

SAL also expects Git for project metadata.
[`sal query`](/sal/reference/commands/#sal-query) opens DuckDB, so install DuckDB if you want to inspect built tables interactively.

## Initialize a repository

Run `sal init` from the root of a Git repository that already has a remote:

```sh
sal init
```

This creates the local SAL data directory and adds `.sal/data` to `.gitignore`.
Generated data is not meant to be committed to source control.

Read more about the generated files in [Data Layout](/sal/reference/data-layout/).

## Add RDF files

SAL finds RDF files in the paths you pass to `sal validate` or `sal build`.
Use Turtle or JSON-LD for source data.

```sh
sal validate data/
```

Validation checks that RDF parses correctly and that prefixed terms resolve against their vocabularies.
If you need to remap vocabulary prefixes while validating, pass one or more `--prefix-maps` values.

## Build the data product

After source changes are committed, build the local Iceberg table:

```sh
sal build data/
```

By default, SAL writes an Iceberg table.
For validation-only workflows or debugging, see [`sal validate`](/sal/reference/commands/#sal-validate) and the `--format` option in [`sal build`](/sal/reference/commands/#sal-build).

## Query the result

Open the built table through DuckDB:

```sh
sal query
```

SAL creates a `triples` view in the DuckDB shell.
From there, run SQL such as:

```sql
SELECT subject, predicate, object
FROM triples
LIMIT 10;
```

Next, publish the generated table using [Publishing Data](/sal/guides/publishing-data/).
